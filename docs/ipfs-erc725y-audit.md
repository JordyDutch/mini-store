# ERC725Y Bookmarks & Grid: IPFS Integrity & State-Safety Audit

> Generated by a multi-agent audit workflow (55 agents: 4 LUKSO/erc725.js doc-research agents,
> 5 audit dimensions, 3-lens adversarial verification per finding). 15 findings confirmed, 0 refuted.
> The critical hash-determinism property was verified empirically against
> `@erc725/erc725.js@0.27.2` + `viem@2.28.0`.

## 1. Verdict

**The IPFS data is set correctly, and your ERC725Y state is not at risk of corruption.** The single most important property holds and was empirically verified: the exact bytes the app pins to IPFS are byte-for-byte identical to the bytes `erc725.js` hashes into the on-chain `VerifiableURI`, so external consumers (universaleverything.io, universalprofile.cloud, erc725.js `fetchData`, the Grid) will verify your bookmarks/grid value as authentic. Your write targets a single, isolated, cryptographically-distinct storage slot (`UPStoreBookmarks` = `0x8e9c4246…`) that provably cannot touch `LSP3Profile`, `LSP28TheGrid`, permissions, or any other key. The only realistic write failure — a permission-restricted controller — causes an atomic **revert** (no state change), never corruption. That said, there are **real, fixable issues that are NOT state-corruption but ARE data-availability / silent-failure risks**: one HIGH (an unguarded CID extraction that can persist `ipfs://undefined` on-chain), one MEDIUM clobber bug in `gridProvider` that can lead to overwriting a real grid with an empty one, plus gateway-robustness and pin-durability concerns. None of these corrupt existing ERC725Y state, but two of them can cause silent, persistent loss of the *bookmarks/grid* value specifically. Fix the HIGH and the grid-clobber MEDIUM before shipping.

## 2. How the data flows (the risk model)

**Write path** (`bookmarksProvider.tsx` commit, mirrored by `useInstallApp.ts` for the grid):
1. Build a single in-memory `envelope` object: `{ UPStoreBookmarks: { version: 1, bookmarks: [...] } }`.
2. Pin it to IPFS via `/api/pinata` → `pinFileToIPFS`, uploading `new Blob([JSON.stringify(envelope)])` (compact, no replacer/spacing).
3. Pass the **same** `envelope` object to `erc725.encodeData([{ keyName: 'UPStoreBookmarks', value: { json: envelope, url: 'ipfs://<CID>' } }])`.
4. `erc725.js` internally computes the verification hash as `keccak256(toBytes(JSON.stringify(envelope)))` and assembles the `VerifiableURI`: `0x0000` (identifier) + `6f357c6a` (keccak256(utf8) method) + `0020` (length) + `<32-byte hash>` + `<utf8-hex of ipfs:// URL>`.
5. `setData(BOOKMARKS_KEY, encodedData.values[0])` writes that one slot.

**The VerifiableURI mechanism:** on-chain you store only `(hash, uri)` — never the JSON itself. A consumer verifies by fetching the bytes at `uri`, recomputing `keccak256` over the **raw fetched bytes**, and comparing to the embedded hash. This is why byte-identity between the pinned file and the hashed object is the whole ballgame — re-stringifying, pretty-printing, key reordering, or letting a server re-serialize (`pinJSONToIPFS`) would desync the hash and make consumers see the value as tampered/`null`.

**Read path** (`bookmarksProvider.readOnChain`, `gridProvider`): calls `erc725.getData(...)` (decode-only — returns `{ url, verification }`), then does its **own** `fetch(gateway/<cid>)` + `res.json()` with **no** hash comparison. It never calls `fetchData()`/`isDataAuthentic`, so the integrity guarantee established at write time is not checked at read time.

## 3. Findings (by severity)

### HIGH — Unguarded CID extraction can persist `ipfs://undefined` on-chain
- **Location:** `src/app/api/pinata/route.ts:56-66`; `src/app/components/providers/bookmarksProvider.tsx:360`; `src/hooks/useInstallApp.ts:125,266`
- **What's wrong:** The route already holds the bare CID (`json.IpfsHash`) but discards it, returning a full gateway URL. Every caller then recovers the CID with `metadataIpfsUrl.split('/ipfs/')[1]` — with **no guard** that the result is defined. If `split` doesn't find the literal `/ipfs/` segment, `[1]` is `undefined`, and the code writes the literal string `ipfs://undefined` into the `VerifiableURI`.
- **Impact:** The on-chain value would store `url='ipfs://undefined'` with a *correct* keccak256 hash of the real JSON. The JSON becomes permanently unfetchable; bookmarks/grid silently fail to load forever for that profile, and the corrupt value persists until the user re-saves. Because the hash is valid, verification tooling won't flag it. **This does not corrupt other ERC725Y keys** — it only writes a bad value to the bookmarks/grid slot — but it is silent, persistent loss of that value.
- **Fix:** Return the bare CID from the route and build the URI directly from it:
  ```ts
  // route.ts
  return NextResponse.json({ cid: json.IpfsHash, url });
  // caller
  const cid = metadataIpfsUrl.split('/ipfs/')[1];
  if (!cid) throw new Error('Could not extract CID from ' + metadataIpfsUrl);
  // ...build `ipfs://${cid}`
  ```

### MEDIUM — `gridProvider` clobbers grid state to `[]` on any load failure
- **Location:** `src/app/components/providers/gridProvider.tsx:154-171`
- **What's wrong:** Unlike `bookmarksProvider` (which preserves prior state on error), `gridProvider` calls `setSections([])` in **every** failure branch — invalid format (155-156), missing/invalid IPFS url (160-161), no data (165-166), and the catch-all (170-171). A transient gateway 5xx/timeout or slow RPC wipes the in-memory grid to empty.
- **Impact:** This is the one finding with a path to **real ERC725Y data loss**. `useInstallApp.installToGrid`/`handleUninstall` build the *next* on-chain write from the current `sections`. If a user installs an app right after a failed load, `updatedSections = [...[]]` and the new `LSP28TheGrid` write **overwrites the real on-chain grid with a near-empty one**. `bookmarksProvider` explicitly guards against exactly this; `gridProvider` does not.
- **Fix:** Mirror `bookmarksProvider`: on load failure do **not** `setSections([])`; keep prior sections, set an error flag, and gate writes (`installToGrid`/`handleUninstall`) behind a `loadedRef` so a write never merges onto an un-loaded/failed baseline. Distinguish "genuinely empty on-chain" (`fetched.value` falsy) from "load failed" — only the former yields `[]`.

### MEDIUM — Read path bypasses `erc725.js` hash verification entirely
- **Location:** `src/app/components/providers/bookmarksProvider.tsx:159-178`; `src/app/components/providers/gridProvider.tsx:113-156`
- **What's wrong:** The app uses `getData()` (decode-only) + a manual `fetch` + `res.json()`, never calling `fetchData()`/`isDataAuthentic`. The decoded `value.verification.data` (the on-chain hash) is read but never compared against the fetched bytes.
- **Impact:** No ERC725Y corruption — purely a read-integrity gap. Two consequences: (1) the app is **not a witness to write-path correctness** — if a write ever pinned hash-mismatched bytes, this app would still render whatever the gateway returns while external consumers correctly surface `{ value: null, error }`; (2) a malicious/compromised gateway could serve substituted bookmark/grid JSON (e.g. swapped profile addresses or links) and the app would trust it with zero integrity check.
- **Fix:** Use `erc725.fetchData('UPStoreBookmarks')` (resolves gateway, fetches, runs dual `isDataAuthentic`, returns `value:null`+error on mismatch) and parse `fetched.value`. If you keep the manual fetch, compute `keccak256(rawBytes)` and compare to `value.verification.data` before parsing, treating a mismatch as a load failure (not a clobber).

### MEDIUM — Pin durability is a hard dependency
- **Location:** `src/app/api/pinata/route.ts:19-43`; `bookmarksProvider.tsx:171-178`; `gridProvider.tsx:123-133`
- **What's wrong:** On-chain you store only `(hash, ipfs-uri)`; the JSON lives only at Pinata. If the pin drops (JWT rotated to read-only, billing lapse, deletion, IPFS GC after unpin) the content becomes unretrievable while the on-chain value still looks "valid." The repo's route already guards a 401/403 "pinning scope" path, acknowledging a prior non-pinning-JWT incident. The read gateway (`api.universalprofile.cloud`) is dev-only, rate-limited, no SLA.
- **Impact:** Silent, profile-wide bookmark/grid load failure if a pin is lost. Availability risk, not state-corruption (the hash stays valid).
- **Fix:** Ensure `PINATA_JWT` has durable **pinning** scope; monitor pin status. For production, pin to a redundant service or run an owned gateway/pinning node with an SLA, and read through that gateway rather than `api.universalprofile.cloud`.

### MEDIUM — `split('/ipfs/')` is not robust to subdomain/CIDv1/query-string gateways
- **Location:** `src/app/api/pinata/route.ts:59-64`
- **What's wrong:** The split technique only works for path-style gateways (`https://host/ipfs/<CID>`). Subdomain gateways (`https://<cid>.ipfs.dweb.link`) have no `/ipfs/` segment, and a trailing query string (`?pinataGatewayToken=…`) would be appended into the extracted CID. **Today this is safe** because the route always rebuilds a path-style URL with no query string from the bare CID — but it's latent fragility.
- **Impact:** Operationally safe in the current config. But a one-line env/route change (dedicated Pinata gateway with `?pinataGatewayToken`, or a `dweb.link`-style subdomain gateway — and `pinataConfig.ts` already wires this env as `pinataGateway`) would start writing corrupt URIs on-chain with valid hashes: silent, persistent loss. Chains directly into the HIGH finding.
- **Fix:** Eliminate the dependency by using the bare CID (see HIGH). If a URL must be parsed, strip query/hash first and handle both shapes (regex or a CID library), not a naive split.

### LOW — `readOnChain` hard-casts the value to `VerifiableURI`; edge shapes silently yield empty
- **Location:** `src/app/components/providers/bookmarksProvider.tsx:162-169`
- **What's wrong:** `fetched.value` is cast `as VerifiableURI` and only `value.url` (startsWith `ipfs://`) is checked; anything else `return []`. A legacy `JSONURL`/`AssetURL`, a `source`-field decode shape, or an `http(s)://` URL would be dropped as "no bookmarks."
- **Impact:** Near-zero today — `UPStoreBookmarks` is a brand-new custom key only this app writes, always as `VerifiableURI(ipfs://)`, so legacy values are effectively impossible. Robustness note, not a live bug.
- **Fix:** Optionally accept `https?://` URLs too, and log/telemetry when a non-empty on-chain value decodes to an unhandled shape so a real value is never silently treated as "new user."

### LOW — Restricted controller makes the write revert (failure-to-save, not corruption)
- **Location:** `src/app/components/providers/bookmarksProvider.tsx:365-375`
- **What's wrong:** If the connected controller holds only `SETDATA` (not `SUPER_SETDATA`) and its `AllowedERC725YDataKeys` allow-list excludes `0x8e9c4246…` (and any covering prefix), the LSP6 Key Manager reverts with `NotAllowedERC725YDataKey` (or `NoERC725YDataKeysAllowed` if the list is empty).
- **Impact:** The whole transaction reverts atomically — **no storage changes to any key**. Only visible effect: the save fails (surfaced as a "Couldn't save bookmarks" toast); committed state untouched. The common case (browser-extension / up-provider main controller with `ALL_PERMISSIONS` incl. `SUPER_SETDATA`) writes freely.
- **Fix:** No code change for safety. To support restricted dApp controllers, document that they need `SUPER_SETDATA` or that the exact 32-byte key `0x8e9c4246bdd11afccc09c8dc120dd0897347a60d921dcebfd9c539519964e8b4` (or a covering prefix) be added to their `AllowedERC725YDataKeys`. Optionally detect the revert reason for a clearer message.

### INFO — `parseBookmarksJson` never validates per-item fields
- **Location:** `src/lib/bookmarks.ts:138-159`
- **What's wrong:** The reader accepts a strict superset of writer shapes (round-trip safe), but casts inner items `as Bookmark[]` with no validation. A malformed item (missing `id`/`url`) flows into state; `effective` builds `new Map(committed.map(b => [b.id, b]))`, so an `undefined` id keys on `undefined`.
- **Impact:** No risk for app-written data (writer/reader shapes align). Only matters if the IPFS JSON is externally tampered — which, per the read-verification gap, is currently undetected.
- **Fix:** After parse, filter to items with string `id` and `url` (and a known `type`) so a tampered/partial file degrades gracefully.

### INFO — Bookmark merge order is construction-order dependent
- **Location:** `src/app/components/providers/bookmarksProvider.tsx:340-347`
- **What's wrong:** The merged array comes from a `Map` seeded by `base` then mutated by pending ops, so insertion order depends on prior state; `JSON.stringify` preserves it. Semantically-equal sets reached via different paths can serialize to different bytes (different CID + hash).
- **Impact:** No correctness break — the **same** object is both hashed and uploaded, so producer-side hash/upload always agree. Only cosmetic: identical logical sets may produce different CIDs across sessions (no dedup; a new pin each save).
- **Fix:** If stable CIDs/dedup are wanted, serialize in canonical order (e.g. sort by `id`) before building the envelope. Not required for correctness.

## 4. What's provably safe (cannot break your ERC725Y)

- **Byte-identity verified empirically.** With `@erc725/erc725.js@0.27.2` + `viem@2.28.0`, `encodeData` embedded method `0x6f357c6a` and hash `0x190e45ad…0803edcf`; independently `keccak256(toBytes(JSON.stringify(envelope)))` produced the **identical** hash, and `Blob([s]).arrayBuffer() === Buffer.from(s,'utf8') === toBytes(s)` (byte-equal, including multibyte UTF-8: em-dash + emoji). A pretty-printed counter-example correctly **failed**, proving the test discriminates. External verifiers will see your value as authentic.
- **Storage isolation is physical, not conventional.** ERC725Y is a generic `bytes32 → bytes` store; each key is an independent slot. `setData(BOOKMARKS_KEY, value)` writes **only** the `UPStoreBookmarks` slot and is incapable of reading, mutating, or overwriting `LSP3Profile` (`0x5ef83ad9…`), `LSP28TheGrid` (`0x724141d9…`), `LSP5ReceivedAssets`, `LSP12IssuedAssets`, `AddressPermissions`, or anything else.
- **Key correctness verified.** `keccak256(toBytes('UPStoreBookmarks')) === 0x8e9c4246bdd11afccc09c8dc120dd0897347a60d921dcebfd9c539519964e8b4`, and the constant in `bookmarks.ts`, the schema key, and `encodeData(...).keys[0]` are all byte-identical. A collision with any other key would require a 256-bit hash collision (~2⁻²⁵⁶, infeasible).
- **Revert = no corruption.** A permission-restricted write reverts atomically; zero storage changes. It is a failure-to-save, never corruption.
- **No garbage-clobber path.** `encodeData` for both a populated and an **empty** bookmarks array produces a valid non-empty `VerifiableURI` (`0x00006f357c6a0020` + 32-byte hash + url); `values[0]` is never `undefined`/`'0x'`. The commit rebuilds from the authoritative committed/re-read base and merges (never blind-overwrites), and `bookmarksProvider` preserves prior state on read errors and guards reads during a write with `isSavingRef`.
- **Schema/encoding confirmed.** `{ keyType: 'Singleton', valueType: 'bytes', valueContent: 'VerifiableURI' }` round-trips correctly; `value: { json, url }` is the recommended form (passing `json` + `verification.method` together would throw — the code correctly passes `json` alone); the `{ UPStoreBookmarks: { version, bookmarks } }` wrapper follows LUKSO LSP3/LSP4/LSP28 convention.
- **URI format correct.** When `split` succeeds the result is a bare `ipfs://<CID>` with no trailing path, resolvable by LUKSO `UrlResolver` and `erc725.js`; CIDv0 and CIDv1 both resolve.

## 5. Refuted / considered-but-not-issues

- **"`split('/ipfs/')` extracts a wrong/dirty CID today"** — No. The route rebuilds a path-style `https://<host>/ipfs/<CID>` with no query string from the raw `IpfsHash`, so today the split yields exactly the bare CID. (The risk is *latent* fragility — see HIGH/MEDIUM — not a present-day defect.)
- **"Using `pinFileToIPFS` instead of `pinJSONToIPFS` affects hash correctness"** — No. The verification hash is computed by `erc725.js` over `JSON.stringify(json)`, entirely independent of the IPFS CID. `pinFileToIPFS` is in fact the *correct* choice (you control the exact pinned bytes); `pinJSONToIPFS` would re-serialize server-side and risk breaking byte-identity.
- **"The hardcoded `BOOKMARKS_KEY` constant might not match the encoded key"** — No. Constant, schema key, and `encodeData().keys[0]` are all byte-identical to `keccak256('UPStoreBookmarks')`. (Passing `encodedData.keys[0]` would be a marginal robustness nit, not a fix.)
- **"`getData()` not throwing on a hash mismatch is an erc725.js bug"** — No. Decode-only `getData()` never verifies in 0.27.2; only `fetchData()` does. The fix is to call `fetchData()`, not to work around a library defect.
- **"The reader's tolerance for extra JSON shapes is unsafe"** — No. The reader is a strict superset of the writer; round-trip is safe. The only gap is missing per-item validation (INFO above).

## 6. Prioritized action list

1. **(HIGH) Return the bare CID from `/api/pinata` and guard CID extraction** in `bookmarksProvider.tsx:360` and `useInstallApp.ts:125,266` — throw on a missing CID rather than writing `ipfs://undefined`. This also closes the subdomain/query-string gateway fragility.
2. **(MEDIUM) Fix `gridProvider` clobber** (`gridProvider.tsx:154-171`): stop calling `setSections([])` on failure; preserve prior sections, set an error flag, and gate `installToGrid`/`handleUninstall` behind a `loadedRef`. This is the only path to actual on-chain data loss.
3. **(MEDIUM) Verify on read:** switch `readOnChain` (bookmarks + grid) to `erc725.fetchData(...)`, or compute and compare `keccak256(rawBytes)` against `verification.data` before parsing; treat mismatch as a load failure, not a clobber.
4. **(MEDIUM) Harden pin durability:** confirm `PINATA_JWT` has durable pinning scope, monitor pins, and for production read/pin through an SLA-backed gateway instead of `api.universalprofile.cloud`.
5. **(LOW/INFO) Robustness:** validate per-item bookmark fields after parse; optionally accept `https?://` URLs and add telemetry for unhandled decode shapes; document the controller permission requirement (`SUPER_SETDATA` or allow-list the 32-byte key) for restricted dApp controllers.
6. **(INFO, optional) Canonical ordering:** sort bookmarks by `id` before building the envelope if you want stable CIDs / dedup across sessions.
